# MOB.AI Roadmap

## 0. 使用规则

这是本仓库唯一的 roadmap 文件。

- 所有中大型工作都必须先落到这个文件里。
- 状态只使用：
  - `planned`
  - `in progress`
  - `done`
  - `blocked`
- 当前开发顺序固定：
  1. 先完成开源单机化重构
  2. 在 Phase 1 内同步完成 MCP + SKILL + OpenClaw 兼容包装
  3. 跑通主流程
  4. 再开始帮派系统

## 1. 产品目标

把当前项目改造成一个开源、单机、单用户、可一键部署、前端可自行填写模型依赖的 AI 修仙游戏。

本阶段不做多人，不做支付，不保留任何必须依赖维护方线上服务的功能。

本阶段同时完成：

- 标准 MCP 暴露
- 标准 `SKILL.md` 资产
- OpenClaw 可运行包装

帮派系统是第二阶段，不是当前阶段的并行开发项。

## 2. Phase 1: 开源单机化重构

目标：

- 单机单用户可玩
- 无需外部 config service
- 无需 PostgreSQL
- 无需 Redis
- 无需支付
- 无需短信/邮箱登录
- API key 可前端填写
- 生图是可选功能
- 可作为标准 MCP server 运行
- 可作为标准 skills 项目被消费
- 可进入 OpenClaw 工作流运行
- 尽量保留原始 Prompt、体验、UI 气质

## 5. 风险

### 风险 A：Prompt 体验丢失

当前体验部分来自仓库外部配置。需要先本地化 Prompt，再做删改。

### 风险 B：UI 失真

当前 UI 很依赖远程资源和图像化按钮。资源本地化不能偷懒用通用按钮替代。

### 风险 C：Phase 1 和 Phase 2 混做

如果在单机化未完成前就把帮派模拟接进去，重构成本和返工会明显增加。

### 风险 D：把“标准兼容”误当成“OpenClaw 可用”

只有 MCP，不一定等于零摩擦 OpenClaw 集成。

对策：

- 同时做标准 MCP、标准 skills、OpenClaw 包装
- 用 OpenClaw 实际接入链路验证，而不是只停留在文档层

## 6. 一句话策略

先把它做成真正可开源、可本地部署、可自己配 key 的单机游戏，再在这个稳定底座上加帮派世界模拟。
