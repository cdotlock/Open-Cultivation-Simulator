# MOB.AI 关系系统 PRD

> 版本：v4（已补关系阶段升级、树状事件池与成人向边界）
> 状态：已落地 MVP，并完成事件深化与阶段扩容
> 适用范围：开源单机版 `MOB.AI`

## 1. 背景

当前仓库已经完成：

- 单机主流程
- 帮派世界上下文
- 世界外循环模拟

但玩家与“重要 NPC”的长期关系仍停留在角色创建时的静态 `人物关系` 文本，缺少：

- 可持续成长的陪伴关系
- 可进入剧情的关键 NPC
- 可独立交互的对话入口
- 可持久化的关系记忆

因此需要补一层统一的 `NPC Bond` 底座，并在其上落地：

- `道侣系统`
- `弟子系统`

## 2. 方案收敛记录

### 2.1 v0 原始想法

原始设想是分别做两套系统：

- 一套弟子系统：金丹后收徒、刷新候选、特性养成、弟子 AI、专属剧情
- 一套道侣系统：筑基后许愿、兑现理想型、恋爱养成、聊天页面、亲密成长

问题：

- 两套系统会重复实现数据结构、记忆、聊天、剧情接入、刷新与 AI 框架
- 后续维护成本高
- Prompt 上下文会出现两套互不兼容的数据格式

### 2.2 v1 中间方案

中间方案尝试直接做一个“关系表”，把所有信息都塞在一张表里。

问题：

- NPC 的静态身份与关系状态耦合过强
- 候选人、已收徒、已结缘、已离场等状态切换不清晰
- 关系记忆与愿望兑现缺少独立载体

### 2.3 v2 最终方案

采用统一 `NPC Bond` 底座：

- `BondActor`：NPC 静态画像
- `CharacterBond`：关系状态与数值
- `BondMemory`：关系记忆摘要
- `BondWish`：道侣愿望与兑现状态

在这个底座上：

- `道侣` 是“单个高深度、强陪伴、强聊天”的 active bond
- `弟子` 是“多个中深度、强养成、强事件”的 active/candidate bond

## 3. 产品目标

### 3.1 核心目标

- 给主流程补上长期关系驱动
- 让关键 NPC 真正参与剧情推进
- 提供独立关系页与聊天页
- 让道侣与弟子具备稳定、可控、可扩展的 AI 感

### 3.2 非目标

- 不做多人社交
- 不做开放世界全镇 25 人自治模拟
- 不做所有 NPC 全量逐回合 LLM 驱动
- 不把关系系统做成独立管理后台

## 4. 设计原则

### 4.1 不破坏主循环

仍保持：

`创建角色 -> 开始剧情 -> 选择行动 -> 检定推进 -> 状态结算 -> 死亡/突破/继续`

关系系统只在主循环上附加：

- 新上下文
- 新入口
- 新事件
- 新聊天交互

### 4.2 代码控结构，LLM 管风味

代码负责：

- 解锁规则
- 槽位规则
- 刷新规则
- 愿望兑现时机
- 关系数值与阶段
- 候选生成骨架
- 记忆存储与裁剪

LLM 负责：

- 道侣/弟子说话方式
- 聊天回复
- 关系事件包装
- 主剧情中的关系侧写

### 4.3 缩放版 Generative Agents

参考 [Generative Agents: Interactive Simulacra of Human Behavior](https://arxiv.org/abs/2304.03442) 的方法，但缩放为“关键 NPC 微代理”：

- observation：记录最近一次聊天/剧情互动
- memory：按摘要存 `BondMemory`
- reflection：达到阈值时提炼关系阶段总结
- planning：只在关键节点生成下一步态度或事件方向

不做逐回合全 NPC 思考。

## 5. 解锁与成长规则

### 5.1 道侣

- 角色从 `炼气` 突破到 `筑基` 且突破成功后，若没有道侣且没有已完成的愿望：
  - 强制进入“许愿道侣”流程
  - 玩家输入想要的理想型
  - 系统保存为 `BondWish`
- 愿望不会立刻兑现，而是在后续若干玩家回合内随机实现
- 同一角色同时只允许一个 active 道侣

### 5.2 弟子

- 达到 `金丹` 后开启收徒资格
- 弟子槽位：
  - 金丹：1
  - 元婴：2
  - 化神：3
  - 炼虚及以上：4
- 候选弟子按周期刷新
- 玩家可从候选中收徒，占用槽位

## 6. 数据模型

### 6.1 BondActor

静态画像层。

关键字段：

- 所属角色 `characterId`
- 类型 `bondType`
- 姓名、称谓、年龄、性别、境界
- 外貌摘要、来历摘要
- 性格标签、公开特性、隐藏特性
- 说话风格
- 喜恶与陪伴调性

### 6.2 CharacterBond

运行态关系层。

关键字段：

- 所属角色 `characterId`
- 绑定 actor `actorId`
- 类型 `bondType`
- lifecycle 阶段 `stage`
- 关系进展阶段 `progressStage`
- 状态 `status`
- 槽位 `slotIndex`
- 数值：`intimacy / trust / loyalty / destiny`
- 当前心情 `mood`
- 下一次事件时机 `nextEventTurn`
- 当前摘要 `summary`

### 6.3 BondMemory

关系记忆层。

关键字段：

- `bondId`
- `sourceType`
- `summary`
- `mood`
- `importance`

### 6.4 BondWish

道侣愿望层。

关键字段：

- `characterId`
- `wishType`
- 原始愿望文本 `rawWish`
- 结构化偏好 `structuredWish`
- 目标出现回合 `targetEncounterTurn`
- `status`

## 7. 阶段机

当前阶段机拆成两层。

### 7.1 lifecycle 阶段

- `WISHING`：已许愿，尚未兑现
- `ACTIVE`：已出现并长期相伴
- `ARCHIVED`：预留，不在本期启用

- `CANDIDATE`：候选弟子
- `ACTIVE`：已收为弟子
- `ARCHIVED`：落选或离场

### 7.2 关系进展阶段

道侣：

- `初识`
- `牵丝`
- `相偎`
- `缠心`
- `同契`

弟子：

- `观望`
- `入门`
- `亲随`
- `得力`
- `倚重`

关系进展阶段由数值阈值决定，并在聊天/事件后自动检查。跨阈值时会额外写入一条 `MILESTONE` 记忆，用来给 UI 和主剧情稳定反馈“关系升级”。

## 8. 候选与兑现机制

### 8.1 道侣兑现

- 愿望写入时生成 `targetEncounterTurn`
- 到达该回合后，如果尚无道侣：
  - 生成一个符合愿望核心偏好的 `BondActor`
  - 建立 `CharacterBond`
  - 在关系页可见
  - 在后续剧情 Prompt 中开始侧写其存在

### 8.2 弟子刷新

- 达到金丹后，每隔若干回合检查一次候选池
- 若当前候选为空或已过冷却：
  - 将旧候选标记归档
  - 生成新候选
- 候选上限暂定为 3

## 9. 事件树与内容边界

### 9.1 树状事件池

事件不再是平铺数组，而是：

- 先按 `bondType`
- 再按 `progressStage`
- 再按 `adultTone / jokeTolerance / desiredScenes / mood`
- 最后从叶子 blueprint 中稳定随机一个事件

这样可以同时做到：

- 道侣在 `牵丝` 以前偏试探和守灯
- 道侣在 `相偎 / 缠心 / 同契` 后更常进入护短、贴近、并肩和克制暧昧
- 弟子在 `入门 / 亲随 / 得力 / 倚重` 后逐渐从闯祸转向争脸面、守门风、替你扛事

### 9.2 玩梗与擦边边界

- 更明显的“擦边”只允许成年道侣线
- 允许的表现是理衣、借伞、贴肩、低声警告、亲手上药、把人拢近、目光停顿
- 禁止露骨色情内容
- 弟子线只增强玩梗、闯祸、护短、求夸和师门烟火气，不写师徒暧昧

## 10. 特性池设计

### 10.1 公共特性

- 天资
- 忠诚
- 口风
- 洁癖
- 财迷
- 爱表现
- 死脑筋
- 记仇

### 10.2 恶搞特性

- 见谁都喊师叔
- 偷喝灵酒
- 写狗屁不通的战报
- 对秘籍只看封面
- 爱把法器起蠢名

### 10.3 轻度成人化表达

仅用于成年道侣与少量成年弟子候选的标签描述，必须是轻度、暧昧、非露骨：

- 爱贴近说话
- 喜欢占座靠肩
- 擅长拿衣袖试探
- 表面清冷，私下黏人

默认只做“标签级表达”，不输出露骨内容。

## 11. AI 与 Prompt 设计

### 11.1 新 Prompt

至少增加：

- `bond_generation_prompt`
- `bond_chat_prompt`
- `bond_event_prompt`

### 11.2 bond_generation_prompt

用途：

- 将道侣愿望文本结构化
- 在关键生成节点补充 NPC 风味化画像

输出：

- 结构化偏好
- NPC 摘要
- 说话风格
- 关系初始摘要

### 11.3 bond_chat_prompt

用途：

- 单独聊天页回复
- 根据最近记忆和当前关系阶段做小幅数值变动

输入：

- 玩家当前状态
- 角色名称
- BondActor 画像
- CharacterBond 当前数值与阶段
- 最近若干条 `BondMemory`
- 用户输入

输出：

- NPC 回复
- 关系摘要更新
- 记忆摘要
- 数值变化建议

### 11.4 bond_event_prompt

用途：

- 把代码侧已经选定的关系事件方向包装成更像剧情钩子的事件摘要
- 给 story prompt 提供可直接引用的关系 hook

输出：

- 事件标题
- 事件摘要
- story hook
- 关系摘要更新
- 记忆摘要
- 小步数值变化建议

## 12. 主剧情接入

### 12.1 Prompt 注入

在 `story_prompt` 注入关系上下文：

- 当前道侣状态
- 当前弟子概况
- 最近关系事件
- 本回合是否存在待兑现或待触发事件
- 本回合优先关系钩子

### 12.2 呈现原则

- 不直接暴露数值
- 通过对话、态度、来信、同路、插话、误会、汇报等方式侧写
- 道侣优先作为长期陪伴出现
- 弟子优先作为候选事件或师徒互动出现

## 13. 页面与交互

### 13.1 角色页

新增轻量关系入口卡：

- 当前有无道侣
- 弟子槽位占用
- 最近关系动向

### 13.2 剧情页

新增轻量关系横条：

- 当前陪伴角色
- 最近师徒/道侣动态
- 可跳转到关系页

### 13.3 关系页

展示：

- 道侣卡
- 愿望输入面板
- 弟子槽位
- 候选弟子列表
- 当前关系事件
- 最近关系记忆

### 13.4 聊天页

展示：

- 关系对象头像区
- 关系数值摘要
- 对话历史
- 输入框

当前 active 弟子与 active 道侣都可以直达聊天页。

## 14. 验收标准

### 14.1 道侣链路

- 新角色能正常推进到突破
- 炼气突破筑基成功后，关系页出现“许愿道侣”
- 愿望提交后，会在若干回合后兑现
- 道侣可在聊天页对话
- 道侣信息能进入主剧情上下文
- 关系事件能产出 story hook 并进入剧情 prompt

### 14.2 弟子链路

- 金丹后开启弟子槽位
- 候选弟子自动刷新
- 可以收徒
- 已收弟子能进入关系页与剧情页
- 已收弟子能直达聊天页，并产出师徒事件

### 14.3 稳定性

- 无关系数据的旧角色不会报错
- Prompt 未配置时软失败，不阻塞主流程
- 所有新增页面移动端可用

## 15. 本期实施边界

本期必须完成：

- 统一 `NPC Bond` 数据底座
- 道侣愿望、兑现、聊天
- 弟子候选、收徒、槽位
- 剧情上下文注入
- 角色页/剧情页/关系页/聊天页

本期不做：

- 离婚、逐出师门、叛逃等复杂分叉
- 多道侣并行
- 弟子独立聊天页的大量玩法
- 跨角色共享关系
