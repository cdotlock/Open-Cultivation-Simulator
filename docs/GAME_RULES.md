# MOB.AI 当前游戏规则档案

> 2026-03-11 更新：
> 开源单机版已经去掉付费复活。当前主流程里的复生规则统一为单机本地复生，不再经过支付链路。

## 1. 核心循环

当前游戏循环可以概括为：

1. 创建角色
2. 生成角色档案与头像
3. 进入角色页
4. 开始剧情
5. 在每一回合从预设选项或自定义输入中做出选择
6. 执行 2d6 检定并推进剧情
7. 扣除行动点，结算状态变化
8. 进入下一回合，直到死亡、触发复活，或理论上进入突破结果

## 2. 角色创建规则

### 2.1 创建输入

玩家当前需要提供或确认：

- 姓
- 名
- 身份
- 灵根
- 三维属性分配：
  - 魅力
  - 神识
  - 身手

### 2.2 手动创建的加点规则

手动创建页的规则来自 `PageCreateChar.tsx`：

- 初始基础点：
  - 魅力 = 1
  - 神识 = 1
  - 身手 = 1
- 额外可分配点数 = 5
- 必须把 5 点全部分完，才能创建
- 每项最终数值上限 = 6
- 灵根会附带额外加成

因此手动创建时：

- 基础三维合计 = 3
- 手动额外点 = 5
- 灵根加成合计 = 2
- 最终三维总和 = 10

### 2.3 灵根加成

| 灵根 | 加成 |
| --- | --- |
| 金灵根 | 神识 +1，身手 +1 |
| 木灵根 | 魅力 +1，神识 +1 |
| 水灵根 | 魅力 +2 |
| 火灵根 | 身手 +2 |
| 土灵根 | 魅力 +1，身手 +1 |

### 2.4 快速开始规则

首页快速开始不是随机点数乱分，而是固定模板：

- 魅力 = 4
- 神识 = 3
- 身手 = 3
- 灵根随机五选一
- 姓名随机
- 身份随机
- 特质从预设列表中随机抽 3 个写入创建提示词

快速开始角色的三维总和同样是 10。

## 3. 角色初始状态

角色创建后，系统会立即生成一个初始 `GamePush`，其状态来自 `getInitialStatus()`：

- 灵根属性：来自角色档案
- 等级：来自 LLM 生成的角色档案初始属性
- 体魄：直接等于当前等级的体魄上限
- 道心：3
- 魅力/神识/身手：来自创建时最终加点
- 行动点：50
- 突破成功系数：0
- 是否死亡：false

## 4. 等级与属性上限

当前境界梯度：

- 炼气
- 筑基
- 金丹
- 元婴
- 化神
- 炼虚
- 合体
- 渡劫
- 真仙

对应上限如下：

| 等级 | 体魄上限 | 道心上限 | 行动点上限 |
| --- | --- | --- | --- |
| 炼气 | 50 | 3 | 50 |
| 筑基 | 70 | 3 | 50 |
| 金丹 | 100 | 3 | 50 |
| 元婴 | 130 | 3.5 | 50 |
| 化神 | 170 | 4 | 50 |
| 炼虚 | 210 | 4.5 | 50 |
| 合体 | 250 | 5 | 50 |
| 渡劫 | 300 | 6 | 50 |
| 真仙 | 350 | 7 | 50 |

## 5. 剧情节点结构

每个 `GamePush` 的 `info` 都符合 `storyPushSchema`，主要分两层：

### 5.1 基础信息

- 类型：`主线 / 机缘 / 危机 / 日常 / 转折`
- 等级：当前境界
- 终极任务是否完成
- 当前任务

### 5.2 剧情要素

- 剧情：正文文本
- 场景：字符串数组
- 人物：字符串数组
- 状态变化：
  - 体魄变化
  - 道心变化
  - 突破成功率变化
  - 行动点变化（schema 有，但当前计算逻辑基本没真正启用）
  - 魅力变化
  - 神识变化
  - 身手变化
- 玩家选项：
  - 选项描述
  - 选项类别
  - 选项难度
  - 预掷骰附加字段

## 6. 回合推进规则

### 6.1 开局

`startGame()` 会读取角色当前状态，调用故事生成 Prompt，产出第一个剧情节点。

### 6.2 常规推进

玩家点击选项后：

1. 前端展示预掷骰结果
2. `pushGame()` 根据选中选项推进剧情
3. 新 `GamePush` 写入数据库
4. 角色 `currentPushId` 指向新节点
5. 后台异步预加载新节点的所有后续选项

### 6.3 自定义输入

玩家可以不选预设选项，而输入自定义行动。

自定义行动流程：

1. 用 LLM 把输入分类成：
  - `交流`
  - `探索`
  - `战斗`
2. 同时判定难度：
  - `轻而易举`
  - `按部就班`
  - `挑战重重`
  - `困难卓绝`
  - `逆天而行`
3. 根据分类后的结果做同样的 2d6 检定
4. 将分析结果一并传给 `pushGame()`

## 7. 选项与检定规则

### 7.1 动作类型

当前只有三类动作：

- 交流
- 探索
- 战斗

### 7.2 难度 DC

| 难度 | DC |
| --- | --- |
| 轻而易举 | 4 |
| 按部就班 | 7 |
| 挑战重重 | 10 |
| 困难卓绝 | 12 |
| 逆天而行 | 14 |

代码里还保留了 `普通 / 困难 / 极困难` 的旧 DC 常量，但当前 schema 不再对外暴露这些难度。

### 7.3 检定公式

当前成功判定公式：

`2d6 + 总修正值 >= DC`

### 7.4 修正值构成

总修正值 = 运气修正 + 能力修正 + 状态修正

#### 运气修正

- 20% 概率：`+1`，理由 `天道眷顾+1`
- 20% 概率：`-1`，理由 `霉运连连-1`
- 60% 概率：`0`

#### 能力修正

公式：对应属性值 `- 3`

| 动作类型 | 对应属性 |
| --- | --- |
| 交流 | 魅力 |
| 探索 | 神识 |
| 战斗 | 身手 |

#### 状态修正

- 体魄 > 35：`神完气足 +1`
- 体魄 < 20 且动作是战斗/探索：`身负重伤 -1`
- 道心 >= 3 且动作是交流/探索：`道心清明 +1`
- 道心 <= 1：`心魔缠身 -1`

### 7.5 预掷骰

当前不是等玩家点击后才计算结果，而是在剧情节点生成后先给每个选项预掷一次：

- 记录成功率
- 记录是否成功
- 记录两个骰子的点数
- 记录基础 DC
- 记录修正值
- 记录修正原因

这意味着玩家点击时看到的是一个已经“预定好结果”的选项。

## 8. 状态变化规则

状态变化由 `attributeSystem.ts` 结算，和直觉不完全一样。

### 8.1 每回合固定规则

无论发生什么：

- 行动点都会 `-1`

### 8.2 成功与失败对状态变化的影响

当前实现里：

- 当 `success === false` 或 `success === undefined` 时，才会处理：
  - 体魄变化
  - 道心变化
- 不论成功失败，都会处理：
  - 突破成功率变化
  - 魅力变化
  - 神识变化
  - 身手变化

这带来两个非常重要的“当前实现事实”：

1. `startGame()` 因为没有传 `success`，所以会按“未成功”分支处理体魄/道心变化。
2. 普通选项如果检定成功，故事里即使声明了体魄/道心变化，当前代码也不会应用。

### 8.3 变化幅度

变化幅度使用枚举：

- 一点
- 中等
- 多
- 极多
- 全部

当前实际映射大致如下：

| 幅度 | 一般属性基础变化 |
| --- | --- |
| 一点 | 2 |
| 中等 | 3 |
| 多 | 5 |
| 极多 | 7 |
| 全部 | 特殊处理 |

特殊点：

- 道心 90% 概率不变化
- 道心变化常量是 0.5 或 1
- 突破成功率变化按等级难度系数换算成 0~1 区间的增减
- 行动点变化字段在 schema 中存在，但当前核心结算逻辑里基本没有使用 Prompt 返回的 `行动点变化`

## 9. 死亡规则

以下任一条件成立，角色会被标记为死亡：

- 体魄 <= 0
- 道心 <= 0
- 行动点 <= 0

死亡后会生成死亡判词，前端跳转到 `/pages/death`。

## 10. 突破规则

### 10.1 触发条件

当前服务层将“突破尝试”绑定在行动点耗尽的时刻：

- 上一状态行动点 = 1
- 新状态行动点 = 0

### 10.2 成功率

突破是否成功取决于：

`Math.random() <= 当前突破成功系数`

### 10.3 成功效果

代码里定义的成功结算会尝试：

- 更新角色描述
- 体魄增加 `极多`
- 道心增加 `一点`
- 行动点意图恢复 `全部`
- 突破成功率增加 `一点`

### 10.4 失败效果

失败时会尝试：

- 更新角色描述
- 体魄增加 `一点`
- 道心增加 `一点`
- 行动点意图恢复 `全部`
- 突破成功率增加 `中等`

### 10.5 当前实现中的关键问题

当前“突破”和“死亡”存在实现顺序冲突：

- `attributeSystem` 会在行动点归零时直接把 `是否死亡` 设为 `true`
- `PageStory.tsx` 收到返回后，先检查 `res.newStatus.是否死亡`
- 一旦为真，前端直接跳死亡页，不再进入突破结果页

所以按现代码推导：

- “行动点耗尽触发突破”是设计意图
- “行动点耗尽直接死亡拦截突破展示”是当前实现结果

这在后续重构时必须明确：到底要保留为规则，还是修成真正的突破结算。

## 11. 复活规则

死亡页文案仍然把复活包装成“服用忆尘丹，保留记忆”，但开源单机版的实际规则已经简化为本地复生。

### 11.1 单机复生

- 不再区分免费复活和付费复活
- 不再经过订单、微信支付、支付回调
- 玩家在死亡页点击“保留记忆”时，直接执行本地复生

### 11.2 复生效果

当前复生不是新开档，而是重置当前推进状态：

- `是否死亡 = false`
- `道心 = 当前等级道心上限`
- `体魄 = 当前等级体魄上限`
- `行动点 = 当前等级行动点上限`

角色档案、剧情链条、当前角色实体都保留。

## 12. 图片生成规则

### 12.1 触发时机

当前开源单机版规则：

- 图像生成默认关闭
- 只有在设置页主动开启后，系统才会检查剧情配图触发
- 当前触发阈值仍然是每推进到第 10 个已选择节点

注意：

- 仓内旧文档写的是“5 局触发”
- 实际代码是 `pushCount >= 10`

### 12.2 生成逻辑

配图 Prompt 会综合：

- 当前 `GamePushResponse`
- 当前角色外貌特征
- `story_illustration` Prompt 配置

然后调用图片模型，并把结果保存到本地 `public/generated/`。

## 13. 叙事控制规则

虽然玩家看到的是“游戏规则”，但大量叙事控制其实在 Prompt 配置层：

- `character_prompt`：角色档案生成
- `story_prompt`：剧情节点生成
- `custom_option_difficulty`：自定义输入分类
- `death_judgement_prompt`：死亡判词
- `breakthrough_success` / `breakthrough_fail`：突破文本与描述更新
- `story_illustration`：剧情配图提示词生成

这些 Prompt 现在已经内置到仓库本地 prompt bundle，并在设置页的二级菜单中维护。

## 14. 当前实现中的规则歧义与重构关注点

### 14.1 行动点变化字段名存实亡

Schema 允许 LLM 返回 `行动点变化`，但主结算逻辑几乎不按这个字段处理，实际只是每回合固定 `-1`。

### 14.2 成功时不结算体魄/道心变化

这属于代码真实规则，但很容易和叙事文本冲突。

### 14.3 突破被死亡分支抢先消费

这是最重要的规则/实现冲突点之一。

### 14.4 预掷骰让选项在点击前已决定成败

这会影响后续如果要做“真正点击时再掷骰”的改造。

## 15. 一句话总结

MOB.AI 当前的规则不是纯数值游戏规则，而是“LLM 生成剧情节点 + 本地检定系统 + 少量硬编码状态机”的混合体。后续重构必须同时处理数值规则、Prompt 契约和前端展现顺序，不能只改其中一层。
